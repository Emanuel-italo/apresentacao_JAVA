Missão · 01 — A solução proposta para a GS

Operar o que orbita, do solo.

A nova corrida tecnológica é a economia espacial: satélites monitoram o clima, orientam o agronegócio, evitam desastres e conectam regiões remotas. O OrbitLink é a plataforma que dá suporte à operação desses ativos — uma API REST que conecta a exploração espacial a problemas reais da Terra.

  • ↳Ciclo de vida dos ativos. Cadastro de satélites, sondas, CubeSats, estações-base e módulos orbitais.
  • ↳Telemetria. Registro de clima, sinal e geolocalização recebidos de cada ativo.
  • ↳Manutenção & alertas. Histórico de intervenções com custo e geração de alertas (colisão, falha, degradação, clima).
  • ↳Missões. Vínculo entre ativos e missões via alocação N:N com chave composta.

Missão · 02 — Arquitetura em camadas

Responsabilidades bem separadas.

Cliente→ Controller→ Service→ Repository→ Banco H2
controllerPorta de entrada REST. Recebe a requisição, valida e devolve apenas DTOs. Nunca expõe entidades.@RestController
serviceRegra de negócio. Orquestra o fluxo, controla transações e converte entidade ↔ DTO.@Service · @Transactional
repositoryAcesso a dados. Interfaces Spring Data JPA que falam com o banco.JpaRepository
entityModelo persistente. Mapeamento JPA das tabelas do domínio.@Entity
dtoContrato da API. Objetos de entrada (Request) e saída (Response) como Java Records imutáveis.record

O pacote ainda traz enums (tipos do domínio), exception (erros + handler global), security (JWT) e config (CORS e carga inicial). Separar camadas mantém o código testável e fácil de evoluir.

Missão · 03 — A utilização do Spring

O ecossistema, peça por peça.

por que entrou no projeto.

CORESpring Boot · autoconfiguração & servidor embarcado›
O que faz: sobe a aplicação com Tomcat embutido e configura tudo por convenção. Por que usamos: elimina XML e boilerplate — `mvn spring-boot:run` e a API está no ar.
WEBSpring Web · controllers REST›
O que faz: mapeia rotas e verbos HTTP com `@RestController` e `ResponseEntity`. Por que usamos: é a base para expor os endpoints e devolver o `HttpStatus` correto.
DATASpring Data JPA · persistência›
O que faz: dá CRUD pronto via `JpaRepository` sobre Hibernate. Por que usamos: escrevemos a interface e o Spring implementa o acesso ao banco — consultas customizadas via JPQL.
SECSpring Security + JWT · autenticação›
O que faz: protege as rotas e valida tokens em cada requisição. Por que usamos: a API é stateless — sem sessão no servidor, autenticação por token assinado.
HATEOASSpring HATEOAS · links hipermídia›
O que faz: anexa links de navegação à resposta com `EntityModel`, `linkTo` e `methodOn`. Por que usamos: leva a API ao nível 3 de maturidade REST — o cliente descobre as próximas ações pela própria resposta.
VALIDSpring Validation · Bean Validation›
O que faz: valida o corpo da requisição com `@Valid` e anotações (`@NotBlank`, `@NotNull`, `@Size`). Por que usamos: rejeita dados inválidos antes da regra de negócio, devolvendo 400 padronizado.
DOCSspringdoc OpenAPI · Swagger UI›
O que faz: gera documentação interativa via `@OpenAPIDefinition`, `@Operation` e `@Tag`. Por que usamos: permite testar os endpoints pelo navegador, com botão Authorize para o token.
DEVLombok + DevTools · produtividade›
O que faz: Lombok gera getters/setters/construtores (`@Data`); DevTools dá hot-reload. Por que usamos: menos código repetitivo e ciclo de desenvolvimento mais rápido.
Missão · 04 — Modelagem avançada

O domínio, em JPA.

Herança — JOINED

AtivoEspacial estende a superclasse abstrata EquipamentoOrbital, que centraliza id, proprietário e número de série.

Por quê: a estratégia JOINED gera uma tabela por classe, normalizando os dados sem repetir colunas comuns.

Embedded

CoordenadasEspaciais (latitude, longitude, altitude) é um @Embeddable embarcado dentro de AtivoEspacial.

Por quê: agrupa campos que andam juntos como um componente reutilizável, sem criar outra tabela.

Chave composta

AlocacaoAtivoMissao usa @EmbeddedId (ativoId + missaoId) e @MapsId para derivar a chave das FKs.

Por quê: representa o relacionamento N:N ativo ↔ missão garantindo uma alocação única por par.

Relacionamentos

Um ativo tem muitas manutenções, telemetrias e alertas (1:N), e participa de várias missões (N:N).

Por quê: modela o mundo real — múltiplas tabelas com integridade referencial via @ManyToOne / @JoinColumn.

No total, 10 entidades compõem o domínio, incluindo Usuario, que implementa UserDetails para integrar com o Spring Security.

Missão · 05 — Persistência

Spring Data JPA & o N+1.

  • ·JpaRepository<T, ID>. Herdar a interface já entrega salvar, buscar, listar e remover — sem implementar nada.
  • ·@Query (JPQL). Consultas customizadas escritas em cima das entidades, não das tabelas.
  • ·join fetch. Traz o ativo junto do alerta na mesma consulta — resolve o problema N+1 e a LazyInitializationException.
  • ·@Modifying. Usado para o update em massa que descomissiona um ativo.

Sem join fetch, listar N alertas dispararia N consultas extras para carregar cada ativo. Uma única consulta resolve tudo.

AlertaOrbitalRepository.java
// CRUD pronto, sem implementação
interface AlertaOrbitalRepository
    extends JpaRepository<AlertaOrbital, Long> {

  // traz o ativo na MESMA consulta
  @Query("select a from AlertaOrbital a " +
         "join fetch a.ativoEspacial " +
         "where a.id = :id")
  Optional<AlertaOrbital> buscarComAtivoPorId(Long id);
}
Missão · 06 — Segurança

Spring Security + JWT, stateless.

1Login. O cliente envia login e senha em POST /api/auth/login.
2Autenticação. O AuthenticationManager valida via DaoAuthenticationProvider + AutenticacaoService.
3Hash. A senha é comparada com o hash BCrypt — nunca armazenamos a senha em texto.
4Token. O TokenService gera um JWT assinado em HMAC256, válido por 2 horas.
5Header. Nas próximas chamadas o cliente envia Authorization: Bearer <token>.
6Filtro. O SecurityFilter (OncePerRequestFilter) valida o token e popula o SecurityContext.

Rotas públicas: /api/auth/login e o Swagger. Todo o resto exige token. Sessão STATELESS e CSRF desabilitado — padrão para APIs REST.

Missão · 07 — A API REST

Endpoints, verbos & status.

POST/api/auth/login›
Autentica e devolve o token JWT. 200 OK.
POST/api/ativos›
Cadastra um ativo e devolve o recurso com links HATEOAS. 201 Created.
GET/api/ativos/{id}›
Busca um ativo por ID, já com o link self. 200 OK ou 404.
POST/api/alertas›
Cria um alerta vinculado a um ativo. 201 Created.
GET/api/alertas/ativo/{ativoId}›
Lista todos os alertas de um ativo específico. 200 OK.
PUT/api/alertas/{id}›
Atualiza um alerta existente. 200 OK.
DELETE/api/alertas/{id}›
Remove um alerta. 204 No Content.

Missão · 09 — Procedimento de teste

Como validamos a solução.

Swagger UI

Login → botão Authorize → colar o token → testar cada endpoint direto pelo navegador.

Valida: contrato, validações e status codes de forma interativa.

cURL & Postman

Fluxo ponta a ponta: obter token → cadastrar ativo → criar alerta → buscar por ID.

Valida: autenticação, CRUD e o caminho feliz completo.

Console H2

Inspeção direta das tabelas em /h2-console para conferir o que foi persistido.

Valida: mapeamento JPA, herança e chaves compostas no banco.

Casos de erro

Buscar id inexistente (404), enviar campos inválidos (400) e token ausente (401/403).

Valida: o GlobalExceptionHandler e as regras de segurança.

Spring Boot Test

O teste contextLoads() garante que o contexto da aplicação sobe sem erros.

Valida: beans, injeção de dependência e configuração.

Deploy no Render

A mesma API rodando em produção (Docker), testável pelo Swagger público.

Valida: o build, o container e o comportamento real em nuvem.
